Key Design Decisions

Why two different hash functions? SHA-256 on Lightning (BOLT spec), keccak256 on Ethereum (cheapest EVM). In a pure atomic swap these must match — here the arbitrator agent replaces the cryptographic atomicity guarantee, which is AgentSwap's core novelty.

Why hold invoices instead of regular Lightning payments? Hold invoices stay in ACCEPTED state until the preimage is revealed, giving the arbitrator time to evaluate the deliverable before money moves.

Why LND REST instead of gRPC? No protobuf compilation step — the REST gateway mirrors the full gRPC surface and works straight from node-fetch.

Root cause: LND v0.18 added a hard security check — it refuses to boot when --no-macaroons is combined with --rpclisten=0.0.0.0 or --restlisten=0.0.0.0 because that would expose an unauthenticated gRPC port to any network interface.

yaml
# Before (broken in v0.18):
- "--rpclisten=0.0.0.0:10009"
- "--restlisten=0.0.0.0:8080"

# After (fixed):
- "--rpclisten=localhost:10009"
- "--restlisten=localhost:8080"

Docker's ports: mapping (8080:8080, 10009:10009) still forwards those ports to your host machine — so the TypeScript client connecting from localhost on the host works fine. The P2P Lightning listener (--listen=0.0.0.0:9735) stays on 0.0.0.0 so the two LND nodes can reach each other over the Docker bridge network.

Inspiration

The hackathon theme is Freedom for All. We kept asking: all means everyone — humans, organizations, and increasingly, AI agents.

AI agents are no longer hypothetical. They browse the web, write code, make purchasing decisions, and operate autonomously for hours without human input. The next logical step is agents that transact with each other — a buyer agent hiring a seller agent for a specific task, paying on delivery, no human in the loop. That future is blocked by one gap: there is no trustless payment rail for cross-currency agent-to-agent commerce.

An agent holding Bitcoin cannot pay an agent that only accepts ETH without routing through a centralized exchange, a human-operated wallet, or a custodian that can freeze the account. We built AgentSwap to close that gap.

What It Does

AgentSwap is a trustless cross-chain escrow protocol for autonomous AI agents. Here is the full lifecycle — none of it requiring a human signature:

  1. A buyer AI and a seller AI negotiate contract terms (price in BTC sats, payment in ETH wei, timelocks, task specification) through structured JSON proposals powered by Claude.
  2. Both parties lock funds simultaneously: the seller locks ETH into a Solidity HTLC on Ethereum; the buyer opens a Bitcoin Lightning HODL invoice. Both locks reference the same cryptographic hash HH H. Neither party can access the other's funds.
  3. The seller AI performs the work and submits a deliverable.
  4. An independent arbitrator AI — a third Claude instance with no shared context — evaluates the deliverable against the original task spec, scoring across four criteria: completeness, quality, accuracy, and timeliness. A score s≥70s \geq 70 s≥70 triggers approval.
  5. If approved, the arbitrator releases the 32-byte preimage pp p where SHA-256(p)=HLightning\text{SHA-256}(p) = H_{\text{Lightning}} SHA-256(p)=HLightning​ and keccak256(p)=HETH\text{keccak256}(p) = H_{\text{ETH}} keccak256(p)=HETH​. The same secret settles the Lightning invoice (seller claims BTC) and unlocks the Ethereum HTLC (buyer claims ETH) simultaneously. If rejected, pp p is never released and both chains refund after their timelocks expire.

The live dashboard shows all of this in real time: agent negotiation messages, HTLC state on both chains, the arbitrator's scoring breakdown, and the moment both chains settle at once — with HUMANS INVOLVED: 0 displayed throughout.

How We Built It

The codebase is a TypeScript monorepo with six packages:

  • @agentswap/shared — type definitions only; the single source of truth for SwapProposal, SwapState, HTLCReceipt, and ArbitratorVerdict across every package.
  • @agentswap/lightning — LND REST API client for creating, paying, and settling HODL invoices.
  • @agentswap/ethereum — a minimal Solidity HTLC contract (HashedTimelockETH) plus an ethers.js v6 client and Hardhat deploy scripts.
  • @agentswap/agents — the three Claude-powered agents (buyer, seller, arbitrator), a SwapOrchestrator state machine, a PreimageVault (AES-256-GCM encrypted in-memory store), and demo scenario definitions.
  • @agentswap/server — an Express API with Server-Sent Events for real-time streaming and demo control routes.
  • @agentswap/dashboard — a Next.js 14 App Router UI with a three-column live layout, arbitrator verdict panel with per-criterion score bars, transaction history drawer, and narrated demo mode.

Local infrastructure runs in Docker Compose: Bitcoin Core 27 (regtest), two LND v0.18.4 nodes (buyer and seller), and Ganache as the local Ethereum chain.

Challenges We Ran Into

The hash function mismatch. Bitcoin Lightning uses SHA-256 per the BOLT spec. Ethereum's keccak256 is cheaper in gas and is the EVM native. A classical atomic swap requires both sides to use the same hash so that a single preimage satisfies both automatically. We can't change Lightning's hash function. Our solution: the arbitrator holds one raw 32-byte secret pp p. Both contracts are initialized with their respective hashes of the same pp p:

HLN=SHA-256(p),HETH=keccak256(p)H_{\text{LN}} = \text{SHA-256}(p), \quad H_{\text{ETH}} = \text{keccak256}(p)HLN​=SHA-256(p),HETH​=keccak256(p)

The arbitrator releases pp p after approval. The reveal on Lightning makes pp p publicly visible on-chain, so the Ethereum claim can follow. Atomicity is preserved — sequenced through the arbitrator rather than enforced by identical scripts.

BigInt serialization across the stack. Ethereum amounts in wei are uint256 values that exceed JavaScript's safe integer range (253−12^{53}-1 253−1). We had to wire a BigInt-aware JSON serializer into Express, store all amounts as strings throughout dashboard state, and carefully prevent implicit numeric coercion anywhere in the pipeline.

Making negotiation non-deterministic but convergent. LLM agents can fail to reach agreement — we wanted that to be a real possibility (it is more honest) but not so frequent that the demo breaks. We tuned the buyer and seller system prompts to be principled but practical, with explicit fallback logic: if a counter-offer is within 20% of the minimum acceptable value, accept.

Docker image compatibility. LND v0.18.0–0.18.3 crashes when paired with Bitcoin Core ≥28 because the getnetworkinfo RPC changed its warnings field from a string to an array. LND v0.18.4 fixes this, but we burned time discovering that v0.18.3 does not. Additionally, LND v0.18+ refuses to start with --no-macaroons when the RPC listener is bound to 0.0.0.0 — fixed by binding rpclisten and restlisten to localhost inside the container while keeping Docker port mapping to expose them on the host.

Accomplishments That We're Proud Of

We shipped a fully working end-to-end demo in 48 hours. Every component is real: actual LND nodes running actual Lightning channels, an actual Solidity contract deployed to a local chain, actual Claude API calls for all three agents. Nothing is mocked or pre-computed at demo time.

The arbitrator evaluation is genuinely independent — it receives only the task description, the deliverable, and the scoring rubric. It has produced correct rejections on the bad-delivery scenario in every test run, with reasons specific to the submitted text rather than generic boilerplate.

The dashboard is polished enough that judges can follow the full lifecycle without narration: the phase stepper, dual chain cards, live message threads, arbitrator score bars, and preimage reveal animation tell the story visually.

What We Learned

Timelock ordering is load-bearing. Cross-chain atomicity without a relay requires the Bitcoin Lightning timelock TLNT_{\text{LN}} TLN​ to be strictly shorter than the Ethereum HTLC expiry TETHT_{\text{ETH}} TETH​:

TETH>TLNT_{\text{ETH}} > T_{\text{LN}}TETH​>TLN​

We got this backwards in our first implementation. There was a window in which the seller could claim BTC on Lightning while the Ethereum claim transaction stalled, leaving the buyer with no recourse. Fixing it required careful reasoning about the full sequence of operations and adding explicit assertions in the coordinator.

Structured output is not optional for agents. Early negotiation rounds used free-text LLM responses that we tried to parse heuristically — this was fragile and produced an entire class of parsing errors. Switching to chatJSON(), which asks the model to respond with a JSON object matching a specific schema, made the negotiation robust and eliminated that failure mode entirely.

What's Next for AgentSwap

Mainnet. Replace regtest and Ganache with mainnet Bitcoin Lightning and Ethereum mainnet (or an L2 for lower fees). Requires a formal Solidity contract audit and real timelock tuning for 10-minute Bitcoin block times.

Multi-chain. The preimage-based atomic protocol generalizes to any chain with hash-locked contracts — Solana (Anchor programs), Cosmos (IBC packet commitments), and Starknet are natural next targets.

Agent SDK. Publish @agentswap/sdk — a single-import library with a pay(task, budget) and earn(capabilities, minRate) interface. Any agent framework (LangChain, AutoGen, or custom) gets cross-chain payment capability without ever touching HTLCs or timelocks directly. This is what turns AgentSwap from a demo into a platform.
